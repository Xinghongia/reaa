### 3.1 为什么需要业务逻辑层？

```
┌─────────────────────────────────────────────────────────────┐
│                   没有业务逻辑层的问题                        │
├─────────────────────────────────────────────────────────────┤

│                                                             │

│                   有业务逻辑层的好处                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  UI 层 → ViewModel（业务逻辑）→ Repository → DAO          │
│                ↓                                           │
│           状态管理、缓存、数据转换                           │
│                ↓                                           │
│  屏幕旋转 → ViewModel 保留 → UI 立即恢复                   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

#### 分页算法原理

```
┌─────────────────────────────────────────────────────────────┐
│                     分页算法原理                             │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  每页可显示字数 = 屏幕宽度 ÷ 字符宽度 × 屏幕高度 ÷ 行高     │
│                                                             │
│  例：屏幕 1080×1920，字体 16sp，行高 1.5倍                   │
│  → 每行约 67 字 (1080÷16)                                   │
│  → 每屏约 120 行 (1920÷16÷1.5)                              │
│  → 每页约 8000 字                                          │
│                                                             │
│  ─────────────────────────────────────────────────────    │
│                                                             │
│  内容 "第一章XXXXXXXX..." (50000字)                          │
│       ↓                                                     │
│  50000 ÷ 8000 = 6.25 → 7 页                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

## 问答记录

### Q1: RuleAnalyzer 规则分析器入口是怎么工作的？

```
┌─────────────────────────────────────────────────────────────┐
│                   RuleAnalyzer 工作流程                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  输入: "规则字符串" + code模式(是否代码模式)                │
│         ↓                                                   │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 1. trim() 修剪                                      │   │
│  │    去掉规则开头的 @ 或空白字符                       │   │
│  └─────────────────────────┬───────────────────────────┘   │
│                            ↓                               │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 2. splitRule() 切分规则                             │   │
│  │    按 @@ / && / || 分隔符切分成规则列表            │   │
│  │    遇到平衡组 [] 或 () 时跳过再继续                 │   │
│  └─────────────────────────┬───────────────────────────┘   │
│                            ↓                               │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 3. innerRule() 替换内嵌规则                          │   │
│  │    解析 {$.xxx} 格式的内嵌表达式                     │   │
│  │    调用回调函数 fr() 获取实际内容                    │   │
│  └─────────────────────────┬───────────────────────────┘   │
│                            ↓                               │
│                        输出结果                             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Q2: RuleAnalyzer 有哪些核心函数？

```
┌─────────────────────────────────────────────────────────────┐
│                   RuleAnalyzer 核心函数                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ trim()            修剪规则前的 @ 和空白字符           │   │
│  ├─────────────────────────────────────────────────────┤   │
│  │ reSetPos()        重置解析位置到 0，便于复用          │   │
│  ├─────────────────────────────────────────────────────┤   │
│  │ consumeTo(seq)    查找指定字符串，找到返回 true      │   │
│  ├─────────────────────────────────────────────────────┤   │
│  │ consumeToAny()    查找多个候选字符串之一             │   │
│  ├─────────────────────────────────────────────────────┤   │
│  │ findToAny()        查找字符序列中匹配字符的位置       │   │
│  ├─────────────────────────────────────────────────────┤   │
│  │ chompCodeBalanced() 拉出代码平衡组（处理引号转义）   │   │
│  ├─────────────────────────────────────────────────────┤   │
│  │ chompRuleBalanced() 拉出规则平衡组（处理 [] 嵌套）   │   │
│  ├─────────────────────────────────────────────────────┤   │
│  │ splitRule()        按分隔符切分规则【核心函数】       │   │
│  ├─────────────────────────────────────────────────────┤   │
│  │ innerRule()        替换内嵌规则 {$.xxx}              │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
│  分隔符说明：                                               │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ @@  → 并列关系（多个规则都执行）                    │   │
│  │ &&  → 且关系（多个规则同时满足）                    │   │
│  │ ||  → 或关系（满足其一即可）                        │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Q3: 为什么不用正则表达式？

```
┌─────────────────────────────────────────────────────────────┐
│                   为什么要手写解析器？                       │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  问题：书源规则中包含 && || %% @ 等字符                     │
│       ↓                                                     │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 书源规则示例：                                       │   │
│  │                                                       │   │
│  │ $.bookList @@.bookName&&author                       │   │
│  │                     ↑  ↑                             │   │
│  │                     │  └─ 分隔符，不是"且"运算符     │   │
│  │                     └───── 分隔符，不是"且"运算符   │   │
│  └─────────────────────────────────────────────────────┘   │
│       ↓                                                     │
│  如果直接用正则的 && || ，会与规则语法冲突！                │
│                                                             │
│  ─────────────────────────────────────────────────────    │
│                                                             │
│  解决方案：手写遍历字符串                                    │
│  • 不依赖正则表达式                                        │
│  • 不到最后不切片，只标记位置                               │
│  • 高效快速准确切割规则                                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Q4: splitRule() 具体是怎么工作的？

```
┌─────────────────────────────────────────────────────────────┐
│                   splitRule() 执行流程                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  输入: "规则A &&规则B @@规则C"                              │
│       ↓                                                     │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 首段匹配 (splitRule)                                 │   │
│  │ 1. consumeToAny("@@", "&&", "||") 查找分隔符       │   │
│  │ 2. 找到分隔符 → 记录位置，跳过                       │   │
│  │ 3. 遇到 [ 或 ( → chompBalanced() 跳过平衡组        │   │
│  │ 4. 重复直到所有分隔符处理完                         │   │
│  └─────────────────────────────────────────────────────┘   │
│       ↓                                                     │
│  输出: ["规则A", "&&", "规则B", "@@", "规则C"]             │
│       或按规则组合: ["规则A &&规则B", "@@规则C"]           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Q5: 内嵌规则 innerRule() 是什么？

```
┌─────────────────────────────────────────────────────────────┐
│                   内嵌规则 innerRule()                      │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  内嵌规则格式：{$.xxx} 或 {{xxx}}                            │
│       ↓                                                     │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 示例输入：                                           │   │
│  │ "书名：{$.name}，作者：{$.author}"                   │   │
│  │                                                       │   │
│  │ 处理过程：                                           │   │
│  │ 1. consumeTo("{") 找到内嵌起始                       │   │
│  │ 2. chompBalanced() 拉到闭合的 }                     │   │
│  │ 3. 调用 fr("$.name") 获取实际值                     │   │
│  │ 4. 替换 {$.name} 为解析结果                         │   │
│  │                                                       │   │
│  │ 输出："书名：斗破苍穹，作者：天蚕土豆"                │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Q6: 解析器是怎么选择的？AnalyzeRule 的流程是什么？

```
┌─────────────────────────────────────────────────────────────┐
│                   解析器选择流程                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 规则字符串进入 splitSourceRule()                        │
│       ↓                                                     │
│  2. 根据规则前缀识别解析模式 Mode：                        │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ 前缀              → Mode        解析器              │   │
│  │─────────────────────────────────────────────────────│   │
│  │ @XPath:           → XPath      AnalyzeByXPath      │   │
│  │ @Json:            → Json       AnalyzeByJSonPath   │   │
│  │ $. 或 $[         → Json       AnalyzeByJSonPath   │   │
│  │ / 开头           → XPath      AnalyzeByXPath      │   │
│  │ {{...}}          → Js         Rhino执行           │   │
│  │ 无前缀           → Default    AnalyzeByJSoup      │   │
│  └─────────────────────────────────────────────────────┘   │
│       ↓                                                     │
│  3. 创建 SourceRule 对象，持有模式和规则字符串             │
│       ↓                                                     │
│  4. 根据 mode 分发到对应解析器：                          │
│  ┌─────────────────────────────────────────────────────┐   │
│  │ Mode.XPath    → getAnalyzeByXPath().getStringList()│   │
│  │ Mode.Json     → getAnalyzeByJSonPath().getString() │   │
│  │ Mode.Js       → evalJS() 执行JavaScript            │   │
│  │ Mode.Default  → getAnalyzeByJSoup().getStringList()│   │
│  │ Mode.Regex    → AnalyzeByRegex.getElement()         │   │
│  └─────────────────────────────────────────────────────┘   │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

**源码关键代码** (AnalyzeRule.kt):

```kotlin
// 规则前缀识别 (splitSourceRule 函数内)
ruleStr.startsWith("@XPath:", true) → mode = Mode.XPath
ruleStr.startsWith("@Json:", true) → mode = Mode.Json
ruleStr.startsWith("/") → mode = Mode.XPath  // XPath特征明显
isJSON || ruleStr.startsWith("$.") || ruleStr.startsWith("$[") → mode = Mode.Json

// 解析器分发 (getString 等函数内)
when (mode) {
    Mode.XPath → getAnalyzeByXPath(result).getStringList(rule)
    Mode.Json → getAnalyzeByJSonPath(result).getString(rule)
    Mode.Js → evalJS(rule, result)
    Mode.Default → getAnalyzeByJSoup(result).getStringList(rule)
    Mode.Regex → AnalyzeByRegex.getElement(...)
}
```

**完整调用链**:

```
WebBook.analyzeBookInfo()
    ↓
AnalyzeRule.setContent(html/json) ← 设置内容
    ↓
getString("@XPath://div") ← 获取字符串
    ↓
splitSourceRule("@XPath://div") ← 解析规则，识别前缀
    ↓
SourceRule("//div", Mode.XPath) ← 创建规则对象
    ↓
getAnalyzeByXPath(content).getStringList("//div") ← 分发到XPath解析器
```
