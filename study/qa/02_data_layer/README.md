## 问答记录

## Q1: 什么是依赖？跟继承有什么区别？

**Q**: 什么是依赖？这跟 Java 中的继承父类有什么区别吗？

**A**:

### 定义对比

| 概念     | 回答的问题     | 关系         |
| ------ | --------- | ---------- |
| **继承** | "A 是什么？"  | A **是** B  |
| **依赖** | "A 需要什么？" | A **使用** B |

### 继承示例

```kotlin
// 继承关系：RssSource "是" BookSource 的一种
open class BookSource { ... }
class RssSource : BookSource() { ... }
```

形成纵向的层级关系：

```
        BookSource（父类）
            ↑
            │
      ┌─────┴─────┐
      │           │
  BookSource   RssSource（子类）
```

### 依赖示例

```kotlin
// 依赖关系：BookRepository "使用" BookDao
class BookRepository(
    private val bookDao: BookDao  // 依赖作为参数传入
) {
    fun getBooks(): Flow<List<Book>> {
        return bookDao.getAll()  // 使用 BookDao 的方法
    }
}
```

形成横向的协作关系：

```
    BookRepository  ──使用──→  BookDao
         │                        ↑
         │                        │
    （我不继承它）              （被注入进来）
```

### 为什么优先用依赖？

**测试场景**：想测试 `BookRepository`，但不想用真实数据库

```
┌─────────────────────────────────────────────────────────────┐
│              继承方式（不好）                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  open class BookRepository {                               │
│      open fun getBooks() = dao.getAll()                    │
│  }                                                         │
│                                                             │
│  class FakeRepository : BookRepository() {                  │
│      override fun getBooks() = fakeData  // 必须重写方法    │
│  }                                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│              依赖方式（好）                                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  class BookRepository(private val bookDao: BookDao)         │
│                                                             │
│  // 测试时直接传入假数据                                     │
│  val fakeDao = FakeBookDao()                               │
│  val repo = BookRepository(fakeDao)  // 传入假的就行        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Legado 项目中的例子

```
┌─────────────────────────────────────────────────────────────┐
│                    Legado 中的实例                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  继承：                                                     │
│  sealed class BookSource { ... }  ← 不同类型的书源"是"书源  │
│  class HttpSource : BookSource()                           │
│  class LocalSource : BookSource()                          │
│                                                             │
│  依赖：                                                     │
│  class BookRepository(                                      │
│      private val bookDao: BookDao,      ← 依赖 DAO          │
│      private val bookSourceDao: BookSourceDao               │
│  )                                                          │
│                                                             │
│  class ReadViewModel(                                       │
│      private val bookRepository: BookRepository  ← 依赖 Repo│
│  ) : ViewModel()                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 总结

| <br />   | 继承              | 依赖              |
| -------- | --------------- | --------------- |
| **关系**   | "是...的一种"       | "使用"            |
| **代码表现** | `class A : B()` | `class A(b: B)` |
| **耦合度**  | 高               | 低               |
| **灵活性**  | 低               | 高               |

## Q2: Koin 的 get() 函数是如何工作的？

**Q**: Koin 的 `get()` 函数是如何工作的？查看代码中 `viewModel { BookViewModel(get()) }`

**A**:

### Koin 的 get() 原理

`get()` 是 Koin 的核心函数，从容器中解析并获取依赖。

### 代码示例

```kotlin
val appModule = module {
    // get() 自动从容器中获取 BookRepository
    viewModel { BookViewModel(get()) }
}

// 容器定义顺序
val appModule = module {
    single { BookRepository(get()) }  // 先定义这个
    viewModel { BookViewModel(get()) }  // get() 找到上面的 BookRepository
}
```

### 工作原理

```
┌─────────────────────────────────────────────────────────────┐
│                    get() 工作流程                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 调用 get<BookRepository>()                            │
│         ↓                                                   │
│  2. Koin 在模块中查找 BookRepository 的定义                 │
│         ↓                                                   │
│  3. 如果是 single { BookRepository(...) }，返回已创建的实例 │
│         ↓                                                   │
│  4. 如果没有，创建一个新实例并缓存（如果是 single）          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 带参数的构造函数

```kotlin
class BookRepository(private val bookDao: BookDao) { ... }

val appModule = module {
    single { BookDao() }  // 提供 BookDao
    single { BookRepository(get()) }  // get() 自动获取 BookDao
}
```

**关键点**：`get()` 会**自动递归解析**所有依赖。

### 懒加载

```kotlin
// 立即获取
val repository: BookRepository = get()

// 懒加载（推荐用于 ViewModel）
val repository: BookRepository by inject()  // 第一次使用时才解析
```

### 总结

`get()` 的工作流程：

1. 从 Koin 容器中查找对应类型
2. 自动解析该类型的所有依赖（递归调用 get()）
3. 返回实例（single 作用域返回同一个实例）

## Q3: 什么叫 Flow 响应式数据流？

**Q**: 什么叫 Flow 响应式数据流？

**A**:

### 什么是"响应式"？

```
┌─────────────────────────────────────────────────────────────┐
│                    现实中的响应式例子                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  📺 电视遥控器                                                │
│  ├── 你按"音量+" → 电视音量增加 → 你看到变化                │
│  └── 你不按 → 电视不变 → 不耗精力去"检查"音量               │
│                                                             │
│  ❌ 非响应式（旧式风扇）                                      │
│  ├── 你要不断伸手去摸风扇叶片检查转速                        │
│  └── 你不知道转速变化，除非伸手去摸                           │
│                                                             │
│  ✅ 响应式（现代风扇）                                        │
│  ├── 你设定转速 → 风扇自动保持                               │
│  └── 转速变化时，风扇自动调整                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 传统方式 vs 响应式

```
┌─────────────────────────────────────────────────────────────┐
│                    传统方式：轮询（Pull）                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  用户打开书架                                                │
│       ↓                                                    │
│  APP：给我书籍列表！（请求）                                │
│       ↓                                                    │
│  数据库：这是当前的书籍列表（响应）                          │
│       ↓                                                    │
│  显示书籍列表                                                │
│       ↓                                                    │
│  用户等待...                                                │
│       ↓                                                    │
│  数据库变了，但 APP 不知道！                                 │
│       ↓                                                    │
│  用户必须手动刷新！                                          │
│                                                             │
│  问题：每次想知道新数据，都要主动"问"一遍                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                    响应式：发布-订阅（Push）                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌──────────┐         ┌──────────┐         ┌──────────┐  │
│  │  数据库   │ ──新数据──→ │  APP     │ ──自动──→ │   UI    │  │
│  │ (发布者)  │         │ (订阅者)  │         │  刷新    │  │
│  └──────────┘         └──────────┘         └──────────┘  │
│       ↑                                                    │
│       │                                                    │
│  数据库变了，                                               │
│  不用你问，                                                 │
│  自动告诉你！                                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Kotlin Flow 是什么？

```
┌─────────────────────────────────────────────────────────────┐
│                      Flow 是什么？                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Flow = 数据的"河流"                                       │
│                                                             │
│  • 生产者（Producer）：产生数据                               │
│  • 消费者（Consumer）：接收数据                               │
│  • 中间操作（Operators）：加工数据                           │
│                                                             │
│  ┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐   │
│  │ 数据库 │──→ │ Flow   │──→ │ map    │──→ │ collect│   │
│  │        │    │        │    │ 加工    │    │        │   │
│  └────────┘    └────────┘    └────────┘    └────────┘   │
│                 数据流               数据流         数据流    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Flow 在 Legado 中的使用

```kotlin
// DAO 层：发布数据
@Dao
interface BookDao {
    @Query("SELECT * FROM books")
    fun getAll(): Flow<List<Book>>  // ← 发布书籍列表
}

// Repository 层：传递数据
class BookRepository(private val bookDao: BookDao) {
    fun getAllBooks(): Flow<List<Book>> = bookDao.getAll()
}

// ViewModel 层：转换数据
class BookShelfViewModel(
    private val bookRepository: BookRepository
) : ViewModel() {
    val books: StateFlow<List<Book>> = bookRepository.getAllBooks()
        .stateIn(viewModelScope, SharingStarted.Lazily, emptyList())
}

// UI 层：消费数据
@Composable
fun BookShelfScreen(viewModel: BookShelfViewModel = viewModel()) {
    val books by viewModel.books.collectAsState()
    //            ↑
    //    订阅 books，当数据变化时自动重组 UI
}
```

```
│  │ StateFlow   │  ← ViewModel 中的 books                      │
│  └──────┬──────┘                                              │
│         ↓ collectAsState()                                     │
│  ┌─────────────┐                                              │
│  │ Compose UI  │  ← recomposition 自动发生                     │
│  └─────────────┘                                              │
│                                                                 │
│  ────────────────────────────────────────────────────────────  │
│                                                                 │
│  用户行为流程：                                                │
│  1. 用户打开书架 → 显示当前书籍                                │
│  2. 后台下载了新章节 → 数据库更新                              │
│  3. Flow 检测到变化 → 自动推送新数据                           │
│  4. Compose 收到新数据 → 自动刷新 UI                           │
│  5. 用户看到最新内容 → 无需手动刷新！                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 对比：Flow vs 普通 List

| <br />   | 普通 List   | Flow         |
| -------- | --------- | ------------ |
| **返回时机** | 一次性返回当前数据 | 数据变化时自动推送新数据 |
| **更新通知** | ❌ 需要手动刷新  | ✅ 自动通知       |
| **线程**   | 同步返回      | 可以在不同线程      |
| **取消**   | 无法取消      | 可以取消订阅       |

### 总结

```
┌─────────────────────────────────────────────────────────────┐

├─────────────────────────────────────────────────────────────┤

│                                                             │

│  好处：                                                    │
│  • 不需要手动刷新                                           │

│                                                             │

```
```
┌─────────────────────────────────────────────────────────────┐

├─────────────────────────────────────────────────────────────┤
```

```
│                                                             │
│  ┌─────────────────────────────────────────────────────┐   │
│  │  Legado：数据访问为主，业务逻辑相对简单               │   │
│  │                                                     │   │
│  │  Repository 主要做：                                 │   │
│  │  • 数据转换（Entity ↔ Model）                        │   │
```

| <br />    | Repository（仓库）             | Service（服务）                   |
| --------- | -------------------------- | ----------------------------- |

| **事务处理**  | 一般不在这里管                    | 通常在这里开启事务                     |
| **业务复杂度** | 简单                         | 复杂                            |

```

│                  DAO vs Mapper 对比                          │

│                                                             │

│  }                                                         │
│                                                             │
│  Room DAO ≈ MyBatis Mapper（都是 ORM 映射层）               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 架构对应关系总结

```
┌─────────────────────────────────────────────────────────────┐
│                  架构对应关系                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Legado (Kotlin)           Java (传统)                     │
│  ─────────────────────────────────────────────────────────  │

│       ↑              ↑            ↑                       │
│       │              │            │                       │

│  ─────────────────────────────────────────────────────────  │
│                                                             │

│                                                             │
│  区别：                                                    │
│  • Java 分层更细，适合复杂业务                              │
│  • Legado 简化后适合数据驱动的阅读器场景                    │
│                                                             │
```

## Q7: Koin 依赖注入是什么？

**Q**: Koin 依赖注入是啥？Koin 是个给安卓 / Kotlin 用的轻量版 Spring IoC 容器，干的事和 Spring 的 Bean 容器一模一样，连核心思想都没换。启动时把类全交给 bean 容器管理，要用就拿，不需要每次都 new。我这样理解对吗？

**A**:

**完全正确！**

### IoC 核心思想

```
┌─────────────────────────────────────────────────────────────┐
│                    IoC 控制权反转                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  传统方式：                                                 │
│  class A {                                                 │
│      fun doSomething() {                                   │
│          val b = B()  // 需要时自己 new                      │
│          b.doIt()                                          │
│      }                                                     │
│  }                                                         │
│                                                             │
│  IoC 方式：                                                │
│  class A(private val b: B) {  // 不 new，靠外面传入         │
│      fun doSomething() {                                   │
│          b.doIt()  // 直接用                                │
│      }                                                     │
│  }                                                         │
│                                                             │
│  控制权反转：A 的控制权交给外部                             │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Koin vs Spring Bean 容器

```
┌─────────────────────────────────────────────────────────────┐
│              Koin vs Spring - 核心一样                       │

│                                                             │
│  Koin                                                      │
│  ─────────────────────────────                             │

│  }                                                         │
│                                                             │
│  共同点：                                                  │
│  • 容器管理 Bean/Singleton                                │

│                                                             │
```

```
│       │                        │                          │
```

## Q10: 书源由哪 6 部分组成？

根据教程《书源规则：从入门到入土》，书源分为 **6 大部分**：

```
┌─────────────────────────────────────────────────────────────┐
│                    书源 6 大组成部分                         │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 基本信息                                                │
│     bookSourceUrl, bookSourceName, bookSourceGroup,         │
│     bookSourceType, enabled, bookUrlPattern, ...            │
│                                                             │
│  2. 搜索配置                                                │
│     searchUrl: String (URL模板)                             │
│     ruleSearch: SearchRule (对象，包含11个字段)              │
│                                                             │
│  3. 发现配置                                                │
│     exploreUrl: String (URL模板)                            │
│     ruleExplore: ExploreRule (对象，包含11个字段)           │
│                                                             │
│  4. 详情页规则                                              │
│     ruleBookInfo: BookInfoRule (对象，包含12个字段)         │
│                                                             │
│  5. 目录规则                                                │
│     ruleToc: TocRule (对象，包含10个字段)                   │
│                                                             │
│  6. 正文规则                                                │
│     ruleContent: ContentRule (对象，包含11个字段)           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 各部分源码对应

| 教程章节 | 源码字段 | 说明 |
|---------|---------|------|
| 基本信息 | `bookSourceUrl`, `bookSourceName`, `bookSourceGroup`, `bookSourceType`, `enabled`... | 书源标识和配置 |
| 搜索 | `searchUrl`, `ruleSearch` | 搜索请求URL和结果解析 |
| 发现 | `exploreUrl`, `ruleExplore` | 发现页面URL和结果解析 |
| 详情页 | `ruleBookInfo` | 书籍详情解析规则 |
| 目录 | `ruleToc` | 章节列表解析规则 |
| 正文 | `ruleContent` | 章节内容解析规则 |

### 哪些字段会参与规则解析？

```
┌─────────────────────────────────────────────────────────────┐
│              参与规则解析的字段（5个规则对象）               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. ruleSearch      → 解析搜索结果页，提取书籍列表          │
│     内部字段：bookList, name, author, intro, coverUrl...   │
│                                                             │
│  2. ruleExplore     → 解析发现/分类页，提取书籍列表        │
│     内部字段：bookList, name, author, intro, coverUrl...   │
│                                                             │
│  3. ruleBookInfo    → 解析详情页，提取书籍信息              │
│     内部字段：name, author, intro, coverUrl, tocUrl...     │
│                                                             │
│  4. ruleToc         → 解析目录页，提取章节列表              │
│     内部字段：chapterList, chapterName, chapterUrl...       │
│                                                             │
│  5. ruleContent     → 解析正文页，提取章节内容              │
│     内部字段：content, subContent, title, nextContentUrl... │
│                                                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│              不参与规则解析的字段（配置/URL类）              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  • 基本信息字段：bookSourceUrl, bookSourceName,             │
│    bookSourceGroup, bookSourceType, enabled...             │
│                                                             │
│  • URL模板字段：searchUrl, exploreUrl,                      │
│    ruleBookInfo.tocUrl, ruleToc.chapterUrl...               │
│                                                             │
│  • 配置字段：enabledExplore, header, jsLib, loginUrl...     │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

> **注意**：规则解析发生在获取到网页/JSON内容之后，以上5个规则对象的内部字段用于从内容中提取数据。URL模板字段只是用来构造请求，不参与内容解析。

## Q16: 连接符号 && || %% 有什么区别？

```
┌─────────────────────────────────────────────────────────────┐
│                   连接符号区别                              │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  符号    名称       行为                                   │
│  ───────────────────────────────────────────────────────   │
│  @@     并列       多个规则都执行，合并结果                │
│  &&     且         合并所有取到的值                        │
│  ||     或         以第一个取到值的为准                    │
│  %%     交替       依次从各列表取值（交叉合并）            │
│                                                             │
│  ───────────────────────────────────────────────────────   │
│                                                             │
│  示例：                                                     │
│  class.name.0@text && class.author.0@text                  │
│  → 先取书名，再取作者，合并在一起                          │
│                                                             │
│  class.name.0@text || class.name.1@text                    │
│  → 取第一个匹配到的书名，如果为空才取第二个                │
│                                                             │
│  name:%%author:%%intro                                     │
│  → [n1, n2] + [a1, a2] + [i1, i2]                         │
│    → [n1, a1, i1, n2, a2, i2]                              │
                                                             │
└─────────────────────────────────────────────────────────────┘
```

## Q17: 书源解析的三个阶段？

书源工作时分为 **3 个阶段**，每个阶段只做一件事：

```
┌─────────────────────────────────────────────────────────────┐
│                  书源解析三阶段流程                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌─────────────┐     ┌─────────────┐     ┌─────────────┐   │
│  │  阶段1       │     │  阶段2       │     │  阶段3       │   │
│  │  构造请求    │ --> │  发起请求    │ --> │  规则解析    │   │
│  │  (URL拼接)   │     │  (网络请求)  │     │  (提取数据)  │   │
│  └─────────────┘     └─────────────┘     └─────────────┘   │
│       ↓                   ↓                   ↓            │
│   URL模板              OkHttp           5大规则对象         │
│   {{key}}              +header          真正干活           │
│   {{page}}             拿原始内容                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 阶段详解

| 阶段 | 做什么 | 用到哪些字段 |
|------|--------|-------------|
| **阶段1：构造请求** | 纯字符串拼接，替换占位符 | `bookSourceUrl` + `searchUrl/exploreUrl/tocUrl` + `{{key}}`, `{{page}}`, `{{$.xxx}}` |
| **阶段2：发起请求** | 引擎硬编码，自动用 OkHttp 发 GET/POST，带上配置的 header | `header`, `enabledCookieJar`...（配置的请求头） |
| **阶段3：规则解析** | 用规则从原始内容中提取数据 | `ruleSearch` / `ruleExplore` / `ruleBookInfo` / `ruleToc` / `ruleContent` |

### 举例说明

以搜索"斗破苍穹"为例：

```
阶段1：URL拼接
  bookSourceUrl  = "http://api.lemiyigou.com"
  searchUrl      = "/search?keyword={{key}}&page={{page}}"
  → 拼接结果："http://api.lemiyigou.com/search?keyword=斗破苍穹&page=1"

阶段2：发起请求
  → OkHttp 自动发送 GET 请求
  → 拿到原始内容：一串 JSON 或 HTML

阶段3：规则解析
  → 把原始内容丢进解析器
  → ruleSearch.bookList = "[*]"         → 提取书籍列表
  → ruleSearch.name = "novelName"       → 提取书名
  → ruleSearch.author = "authorName"    → 提取作者
```

> **注意**：阶段1和阶段2完全不碰规则，只有阶段3才用到5大规则对象。
