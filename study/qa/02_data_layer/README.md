## 问答记录

## Q1: 什么是依赖？跟继承有什么区别？

**Q**: 什么是依赖？这跟 Java 中的继承父类有什么区别吗？

**A**:

### 定义对比

| 概念 | 回答的问题 | 关系 |
|------|-----------|------|
| **继承** | "A 是什么？" | A **是** B |
| **依赖** | "A 需要什么？" | A **使用** B |

### 继承示例

```kotlin
// 继承关系：RssSource "是" BookSource 的一种
open class BookSource { ... }
class RssSource : BookSource() { ... }
```

形成纵向的层级关系：
```
        BookSource（父类）
            ↑
            │
      ┌─────┴─────┐
      │           │
  BookSource   RssSource（子类）
```

### 依赖示例

```kotlin
// 依赖关系：BookRepository "使用" BookDao
class BookRepository(
    private val bookDao: BookDao  // 依赖作为参数传入
) {
    fun getBooks(): Flow<List<Book>> {
        return bookDao.getAll()  // 使用 BookDao 的方法
    }
}
```

形成横向的协作关系：
```
    BookRepository  ──使用──→  BookDao
         │                        ↑
         │                        │
    （我不继承它）              （被注入进来）
```

### 为什么优先用依赖？

**测试场景**：想测试 `BookRepository`，但不想用真实数据库

```
┌─────────────────────────────────────────────────────────────┐
│              继承方式（不好）                                │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  open class BookRepository {                               │
│      open fun getBooks() = dao.getAll()                    │
│  }                                                         │
│                                                             │
│  class FakeRepository : BookRepository() {                  │
│      override fun getBooks() = fakeData  // 必须重写方法    │
│  }                                                         │
│                                                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│              依赖方式（好）                                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  class BookRepository(private val bookDao: BookDao)         │
│                                                             │
│  // 测试时直接传入假数据                                     │
│  val fakeDao = FakeBookDao()                               │
│  val repo = BookRepository(fakeDao)  // 传入假的就行        │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Legado 项目中的例子

```
┌─────────────────────────────────────────────────────────────┐
│                    Legado 中的实例                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  继承：                                                     │
│  sealed class BookSource { ... }  ← 不同类型的书源"是"书源  │
│  class HttpSource : BookSource()                           │
│  class LocalSource : BookSource()                          │
│                                                             │
│  依赖：                                                     │
│  class BookRepository(                                      │
│      private val bookDao: BookDao,      ← 依赖 DAO          │
│      private val bookSourceDao: BookSourceDao               │
│  )                                                          │
│                                                             │
│  class ReadViewModel(                                       │
│      private val bookRepository: BookRepository  ← 依赖 Repo│
│  ) : ViewModel()                                           │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 总结

| | 继承 | 依赖 |
|--|------|------|
| **关系** | "是...的一种" | "使用" |
| **代码表现** | `class A : B()` | `class A(b: B)` |
| **耦合度** | 高 | 低 |
| **灵活性** | 低 | 高 |

## Q2: Koin 的 get() 函数是如何工作的？

**Q**: Koin 的 `get()` 函数是如何工作的？查看代码中 `viewModel { BookViewModel(get()) }`

**A**:

### Koin 的 get() 原理

`get()` 是 Koin 的核心函数，从容器中解析并获取依赖。

### 代码示例

```kotlin
val appModule = module {
    // get() 自动从容器中获取 BookRepository
    viewModel { BookViewModel(get()) }
}

// 容器定义顺序
val appModule = module {
    single { BookRepository(get()) }  // 先定义这个
    viewModel { BookViewModel(get()) }  // get() 找到上面的 BookRepository
}
```

### 工作原理

```
┌─────────────────────────────────────────────────────────────┐
│                    get() 工作流程                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  1. 调用 get<BookRepository>()                            │
│         ↓                                                   │
│  2. Koin 在模块中查找 BookRepository 的定义                 │
│         ↓                                                   │
│  3. 如果是 single { BookRepository(...) }，返回已创建的实例 │
│         ↓                                                   │
│  4. 如果没有，创建一个新实例并缓存（如果是 single）          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 带参数的构造函数

```kotlin
class BookRepository(private val bookDao: BookDao) { ... }

val appModule = module {
    single { BookDao() }  // 提供 BookDao
    single { BookRepository(get()) }  // get() 自动获取 BookDao
}
```

**关键点**：`get()` 会**自动递归解析**所有依赖。

### 懒加载

```kotlin
// 立即获取
val repository: BookRepository = get()

// 懒加载（推荐用于 ViewModel）
val repository: BookRepository by inject()  // 第一次使用时才解析
```

### 总结

`get()` 的工作流程：
1. 从 Koin 容器中查找对应类型
2. 自动解析该类型的所有依赖（递归调用 get()）
3. 返回实例（single 作用域返回同一个实例）

## Q3: 什么叫 Flow 响应式数据流？

**Q**: 什么叫 Flow 响应式数据流？

**A**:

### 什么是"响应式"？

```
┌─────────────────────────────────────────────────────────────┐
│                    现实中的响应式例子                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  📺 电视遥控器                                                │
│  ├── 你按"音量+" → 电视音量增加 → 你看到变化                │
│  └── 你不按 → 电视不变 → 不耗精力去"检查"音量               │
│                                                             │
│  ❌ 非响应式（旧式风扇）                                      │
│  ├── 你要不断伸手去摸风扇叶片检查转速                        │
│  └── 你不知道转速变化，除非伸手去摸                           │
│                                                             │
│  ✅ 响应式（现代风扇）                                        │
│  ├── 你设定转速 → 风扇自动保持                               │
│  └── 转速变化时，风扇自动调整                                │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 传统方式 vs 响应式

```
┌─────────────────────────────────────────────────────────────┐
│                    传统方式：轮询（Pull）                    │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  用户打开书架                                                │
│       ↓                                                    │
│  APP：给我书籍列表！（请求）                                │
│       ↓                                                    │
│  数据库：这是当前的书籍列表（响应）                          │
│       ↓                                                    │
│  显示书籍列表                                                │
│       ↓                                                    │
│  用户等待...                                                │
│       ↓                                                    │
│  数据库变了，但 APP 不知道！                                 │
│       ↓                                                    │
│  用户必须手动刷新！                                          │
│                                                             │
│  问题：每次想知道新数据，都要主动"问"一遍                    │
│                                                             │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│                    响应式：发布-订阅（Push）                 │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  ┌──────────┐         ┌──────────┐         ┌──────────┐  │
│  │  数据库   │ ──新数据──→ │  APP     │ ──自动──→ │   UI    │  │
│  │ (发布者)  │         │ (订阅者)  │         │  刷新    │  │
│  └──────────┘         └──────────┘         └──────────┘  │
│       ↑                                                    │
│       │                                                    │
│  数据库变了，                                               │
│  不用你问，                                                 │
│  自动告诉你！                                               │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Kotlin Flow 是什么？

```
┌─────────────────────────────────────────────────────────────┐
│                      Flow 是什么？                           │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Flow = 数据的"河流"                                       │
│                                                             │
│  • 生产者（Producer）：产生数据                               │
│  • 消费者（Consumer）：接收数据                               │
│  • 中间操作（Operators）：加工数据                           │
│                                                             │
│  ┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐   │
│  │ 数据库 │──→ │ Flow   │──→ │ map    │──→ │ collect│   │
│  │        │    │        │    │ 加工    │    │        │   │
│  └────────┘    └────────┘    └────────┘    └────────┘   │
│                 数据流               数据流         数据流    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### Flow 在 Legado 中的使用

```kotlin
// DAO 层：发布数据
@Dao
interface BookDao {
    @Query("SELECT * FROM books")
    fun getAll(): Flow<List<Book>>  // ← 发布书籍列表
}

// Repository 层：传递数据
class BookRepository(private val bookDao: BookDao) {
    fun getAllBooks(): Flow<List<Book>> = bookDao.getAll()
}

// ViewModel 层：转换数据
class BookShelfViewModel(
    private val bookRepository: BookRepository
) : ViewModel() {
    val books: StateFlow<List<Book>> = bookRepository.getAllBooks()
        .stateIn(viewModelScope, SharingStarted.Lazily, emptyList())
}

// UI 层：消费数据
@Composable
fun BookShelfScreen(viewModel: BookShelfViewModel = viewModel()) {
    val books by viewModel.books.collectAsState()
    //            ↑
    //    订阅 books，当数据变化时自动重组 UI
}
```

```
│  ┌─────────────┐                                              │
│  │ StateFlow   │  ← ViewModel 中的 books                      │
│  └──────┬──────┘                                              │
│         ↓ collectAsState()                                     │
│  ┌─────────────┐                                              │
│  │ Compose UI  │  ← recomposition 自动发生                     │
│  └─────────────┘                                              │
│                                                                 │
│  ────────────────────────────────────────────────────────────  │
│                                                                 │
│  用户行为流程：                                                │
│  1. 用户打开书架 → 显示当前书籍                                │
│  2. 后台下载了新章节 → 数据库更新                              │
│  3. Flow 检测到变化 → 自动推送新数据                           │
│  4. Compose 收到新数据 → 自动刷新 UI                           │
│  5. 用户看到最新内容 → 无需手动刷新！                          │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

### 对比：Flow vs 普通 List

| | 普通 List | Flow |
|--|----------|------|
| **返回时机** | 一次性返回当前数据 | 数据变化时自动推送新数据 |
| **更新通知** | ❌ 需要手动刷新 | ✅ 自动通知 |
| **线程** | 同步返回 | 可以在不同线程 |
| **取消** | 无法取消 | 可以取消订阅 |

### 总结

```
┌─────────────────────────────────────────────────────────────┐
│                        总结                                  │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Flow = 响应式数据流                                         │
│                                                             │
│  • 你订阅（collect）一个数据源                               │
│  • 数据源变化时，自动推送新数据给你                          │
│  • 你收到数据后，自动更新 UI                                │
│                                                             │
│  好处：                                                    │
│  • 不需要手动刷新                                           │
│  • 数据实时同步                                             │
│  • 代码更简洁                                               │
│                                                             │
│  在 Legado 中的流程：                                       │
│  数据库变化 → Flow 发射 → Repository 传递 → ViewModel 转换  │
│       → Compose 订阅 → UI 自动刷新                          │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```
